# デザイン案と実装の優先方針（大前提）

最終更新: 2026-05-18

## 背景

- Figma および `Lobby 初期UI.pdf` などの**デザイン案は、Cursor で実装仕様を固める前に作られた**ものである。
- その後、Cursor 上の指示・仕様メモ・Firestore ルールに沿って**機能・画面・文言が追加・変更されている**。
- マイページ整備など「デザインに寄せる」作業をしても、**既に動いている実装をデザインに合わせて壊さない**ことが前提である。

## 優先順位（矛盾したとき）

| 優先度 | 正本 | 備考 |
|--------|------|------|
| 1 | `firestore.rules` / `lib/lobby-firestore-types.ts` | データ・権限 |
| 2 | `LOBBY_FINAL_SPEC.md` | 機能・運用の完成系 |
| 3 | **既にマージ済みの実装**（本リポジトリのコード） | Cursor で作った挙動・表示 |
| 4 | `UI_UX_SPEC.md` | 記録用。PDF と実装がずれている箇所あり |
| 5 | Figma / PDF / スクショ | **見た目・不足 UI の参考**。実装と矛盾する場合は 3 を維持 |

**結論: Figma より Cursor（＝現行実装・仕様メモ）を優先する。**

スクショや Figma から UI を直すときは、「デザインに合わせて既存機能を削る・変える」のではなく、**デザインで足りない見た目を足し、既存の挙動は残す**方向にする。

---

## 変えてはいけないもの（原則・維持リスト）

デザイン作業の対象外。スクショ実装の提案に含めない。

- **認証・オンボーディング**: 本人確認・チケットゲート、`admins/{uid}` による運営スキップ、停止アカウント画面
- **参加者番号**: 初回ダッシュボード入場時付与、No.001〜（1000以降4桁）、運営 No.000、`lib/format-participant-no.ts`
- **ホーム**: QR「表示する」／カメラ「スキャン」／コード入力モーダル、シーズン帯・カウントダウン、お知らせ折りたたみ（ベル・未読バッジ）、運営のみ「管理者でログイン中」
- **ボトムナビ**: 左から **履歴 → イベント → ホーム（やや大）→ チャット → マイページ**（5タブ）。お知らせは独立タブにしない
- **履歴タブ**: マッチ一覧・ブロック・通報（コード手入力はホーム側）
- **チャット**: 1:1 DM・24h/72h 解放・運営は期限なし（ボード投稿は廃止済み）
- **イベント・マッチング・通報・Functions** など `LOBBY_FINAL_SPEC.md` に書いたデータモデルと API
- **UID・マッチングコードの露出方針**（マイページ等で UID を出さない等、既に決めた表示ルール）
- **開発用バイパス帯**: 運営（`admins`）には出さない。env ベースの開発バイパスのみ

上記は随時 `LOBBY_FINAL_SPEC.md` とコードが更新される。**疑問があればコードと FINAL SPEC を見る。**

## 変えてよいもの（デザイン寄せの典型）

- 対象画面の**レイアウト・余白・タイポ・色のトークン**（`docs/THEME_COLORS.md` の `--lobby-red` / `--lobby-cream` の範囲）
- **まだ仮のままの UI**（例: マイページのカード構成・見出し・ボタン配置）
- デザインにあって**仕様にない新規の静的ラベル**（既存フィールドを消さない範囲）
- アイコン・イラストの差し替え（ナビの PNG パスやサイズ方針は、合意後にのみ変更）

## 作業フロー（一気に変えない）

Figma・PDF・スクショに基づく**画面単位のまとめ直し**を行う前に、必ず次を行う。

### 1. 差分一覧を書く（ユーザー確認用）

チャットまたは PR 説明に、次の形式で出す。

```markdown
## [画面名] デザイン反映 — 確認用

### 変えない（現行実装を維持）
- …
- …

### 変える（デザイン／スクショに合わせる）
- …
- …

### 要確認（デザインと実装が両立しない）
- …
```

- **変えない** … 上記「維持リスト」＋その画面で既に動いている機能
- **変える** … 見た目のみ・仮 UI の置き換え
- **要確認** … デザインどおりにすると既存仕様と衝突する項目（選択肢を添える）

### 2. ユーザーが OK してから実装

「一覧どおりで進めてよい」の返信、または項目ごとの修正指示をもらってからコーディングする。

### 3. 実装後

- 触ったファイルと、意図的に**あえて変えなかった**点を短く報告する。
- `UI_UX_SPEC.md` を更新する場合は、**実装に合わせて**書く（PDF 文言のままに戻さない）。

---

## マイページ整備について（次タスク）

- 素材: **スクショ**（Figma MCP は使わない想定で可）
- 手順: スクショ受領 → **上記「差分一覧」で最終確認** → 実装
- 正本: 表示項目・編集可否は `DashboardProfileSection` / Firestore プロフィールフィールドと `LOBBY_FINAL_SPEC.md` に従う
